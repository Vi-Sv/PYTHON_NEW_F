## 1. Введение в архитектуру: Закрытие конвейера обработки данных
До этого момента мы импортировали, чистили, фильтровали и агрегировали данные внутри изолированного бункера оперативной памяти (RAM) с помощью Pandas. Но RAM энергозависима: скрипт завершил работу — данные испарились. Финальная стадия любого промышленного конвейера аналитика — **перманентная консервация результатов**. 

Вместо того чтобы вручную выгружать таблицы в Excel-файлы и захламлять жесткий диск, мы наводим прямой скоростной мост между Python и промышленной базой данных СУБД (PostgreSQL / MySQL) с помощью библиотеки **SQLAlchemy**.

* **В ручном режиме:** Аналитик сохраняет CSV, открывает клиент СУБД (DBeaver/pgAdmin), создает таблицу руками и импортирует файл.
* **В автоматическом режиме:** Скрипт Python одной командой создаёт таблицу нужной структуры в базе данных и заливает туда миллионы строк за секунды.

---

## 2. Инструмент №1: Движок подключения — `create_engine()`

### Логика инструмента
Инструмент библиотеки SQLAlchemy, который создает постоянную точку связи («туннель») между скриптом Python и сервером базы данных. Через этот туннель транслируются SQL-запросы и передаются пакеты строк.

### Синтаксис
```python
from sqlalchemy import create_engine

# Сборка строки подключения (URL-адрес базы данных)
# Формат: СУБД://Логин:Пароль@Сервер:Порт/Имя_Базы
db_url = "postgresql://pi_operator:Secr3tPass@192.168.1.50:5432/production_db"

# Инициализация движка-туннеля
engine = create_engine(db_url)
```

### Физика работы
При вызове `create_engine()` Python не подключается к базе мгновенно. Он лишь формирует чертеж подключения в RAM. Физический контакт и аутентификация логина/пароля произойдут автоматически, когда мы попытаемся отправить первую порцию данных.

---

## 3. Инструмент №2: Автоматический десант в СУБД — метод `.to_sql()`

### Логика инструмента
Встроенный в Pandas метод, который берет готовый `DataFrame` из оперативной памяти ПК, автоматически сканирует его структуру (названия колонок и типы данных), создает идентичную физическую таблицу на сервере СУБД и осуществляет пакетную заливку строк.

### Синтаксис
```python
# Настоящий приказ: "Сформировать таблицу в СУБД и залить туда данные из RAM"
df_report.to_sql(
    name="vitrina_payout_smr",  # Физическое имя новой таблицы в базе данных
    con=engine,                  # Наш созданный движок-туннель подключения
    if_exists="replace",        # Стратегия поведения, если таблица уже есть
    index=False,                # Отрубить технический индекс Pandas (Закон!)
    chunksize=10000             # Размер пакета для скоростной загрузки
)
```

### Физика работы: Стратегии параметра `if_exists`
Это рубильник, определяющий поведение системы при столкновении с существующей таблицей в базе:
* `if_exists='fail'` — По умолчанию. Если таблица с таким именем уже создана в СУБД, скрипт мгновенно прекратит работу с ошибкой `ValueError`, защищая данные от перезаписи.
* `if_exists='replace'` — Жесткий приказ «Снести и пересоздать». Система выполняет в базе команду `DROP TABLE`, уничтожая старый архив, создает чистую структуру заново и заливает свежие данные.
* `if_exists='append'` — Режим дозаписи (хронология). Система оставляет структуру старой таблицы нетронутой и просто приклеивает новые строки вниз (аналог вертикального моста `UNION ALL` или команды `INSERT INTO`).

### Капкан №1: Попытка залить DataFrame с грязными именами колонок
Если в твоем `DataFrame` названия колонок написаны на русском языке, содержат пробелы или спецсимволы (например, `"Объем поставки (тн)"`), Pandas честно попытается создать таблицу с такими именами в СУБД. Но если для PostgreSQL русские буквы в кавычках допустимы, то при попытке сделать к этой таблице SELECT в СУБД, ты столкнешься с постоянной необходимостью зажимать имена в двойные кавычки. Скрипты автоматизации начнут ломаться.

### Защита Мастера
Перед вызовом метода `.to_sql()` всегда принудительно переводи все заголовки колонок датафрейма в нижний регистр, заменяй пробелы на нижнее подчеркивание `_` и избавляйся от кириллицы. Колонки должны выглядеть монолитно: `volume_tons`, `line_id`, `delivery_status`.

### Капкан №2: Зависание сервера при построчной заливке больших массивов
Если в твоем датафрейме 1 000 000 строк, и ты вызовешь метод `.to_sql()` без дополнительных настроек, Pandas начнет отправлять на сервер базы данных миллион отдельных команд `INSERT`. Сетевой трафик захлебнется, процессор СУБД уйдет в 100% нагрузку, и заливка растянется на часы.

### Защита Мастера — Пакетная нарезка через `chunksize`
Всегда явно прописывай параметр `chunksize=10000` внутри метода `.to_sql()`. Это приказ системе: «Нарезать массив в RAM на пакеты по 10 000 строк и отправлять их в базу единым залпом». Скорость десантирования данных вырастает в сотни раз.

## 4. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (КОД ДЛЯ ТВОЕГО ВЕРСТАКА)

Этот код полностью автономен и готов к тестам без развертывания тяжелого внешнего сервера PostgreSQL. Мы используем встроенный в Python легковесный движок **SQLite**, который создает полноценную базу данных прямо в файле на твоем жестком диске. 

Скрипт имитирует завершение рабочего дня аналитика: берет очищенную в Pandas витрину данных табеля ПТО и автоматически выгружает её на вечное хранение в локальную СУБД, формируя готовую физическую таблицу.

```python
# =====================================================================
# СЦЕНАРИЙ: Автоматическая консервация аналитической витрины ПТО в СУБД
# =====================================================================

import pandas as pd
from sqlalchemy import create_engine

print("--- ИНИЦИАЛИЗАЦИЯ ТРАНСПОРТНОГО КОНВЕЙЕРА В СУБД ---")

# 1. Готовый очищенный DataFrame в RAM, прошедший все стадии рефакторинга
clean_vitrina_data = {
    "worker_id":,
    "worker_name": ["ИВАНОВ И.И.", "СИДОРОВ А.П.", "НИКОЛАЕВ К.Д.", "ПЕТРОВ В.С."],
    "hours_fact":,
    "payout_rub": [4000.0, 3500.0, 4000.0, 2000.0],
    "sync_status": ["PROCESSED", "PROCESSED", "PROCESSED", "PROCESSED"]
}
df_to_save = pd.DataFrame(clean_vitrina_data)

# 2. Создаем движок-подключение к локальной базе данных SQLite
# Файл 'pto_archive.db' автоматически родится в папке со скриптом
engine = create_engine("sqlite:///pto_archive.db")

print("[СИНХРОНИЗАЦИЯ] Наведение моста с базой pto_archive.db выполнено успешно.")

# 3. Десантируем данные из оперативной памяти в физическую таблицу СУБД
# Применяем стратегию 'replace': если таблица существовала — сносим старую и пишем свежую
df_to_save.to_sql(
    name="vitrina_daily_payout", 
    con=engine, 
    if_exists="replace", 
    index=False, 
    chunksize=5000
)

print("[УСПЕХ] Данные успешно законсервированы в таблицу 'vitrina_daily_payout'.")
print("-" * 60)

# 4. КОНТРОЛЬНАЯ ПРОВЕРКА: Проверяем, физически ли легли данные в базу?
# Отправляем прямой SQL-запрос в созданную базу данных через инструмент Pandas pd.read_sql
print("[АУДИТ ХРАНИЛИЩА] Делаем контрольный SELECT из СУБД для проверки:")

df_check_from_db = pd.read_sql("SELECT * FROM vitrina_daily_payout WHERE hours_fact > 5", con=engine)

print(df_check_from_db)
print("\n--- ТРАНСПОРТНЫЙ КОНВЕЙЕР УСПЕШНО ЗАКРЫТ. СИСТЕМА СТАБИЛЬНА ---")
```
