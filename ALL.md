# Глава 1. ИНИЦИАЛИЗАЦИЯ И ЯДРО ЯЗЫКА PYTHON: ПЕРЕМЕННЫЕ, БАЗОВАЯ МАТЕМАТИКА И ПЕРЕПЛАВКА ТИПОВ

## 1. Введение в архитектуру: Python как среда манипуляции данными
В мире SQL данные покоятся на изолированных островах-таблицах, а СУБД выполняет приказ, возвращая проекцию. В Python мир устроен иначе: здесь нет жестких физических ячеек, пока мы сами их не создадим. Python для аналитика — это высокоскоростной конвейер в оперативной памяти (RAM), где данные непрерывно текут через трубы функций, переплавляются из одного формата в другом и выплескиваются в финальный отчет.
* **Таблица в SQL** — это монолитная плита.
* **Скрипт в Python** — это сборочный цех, где данные находятся в движении. 

---

## 2. Инструмент №1: Вывод данных на экран — `print()`
Базовый инструмент ведения логов и визуализации состояния системы на определенном этапе выполнения конвейера.

### Логика инструмента
`print()` не меняет данные, он копирует текущее значение переменной или текст из RAM и швыряет его в консоль (на экран) для визуального контроля оператором.

### Синтаксис
```python
print("Инициализация системы ПТО выполнена.")
print(variable_name)
```

### Физика работы
Система берет указанный объект, переводит его в текстовый формат на лету и отправляет в стандартный поток вывода. Перечисление элементов через запятую внутри `print()` автоматически склеивает их через пробел.

### Капкан №1: Попытка вывести несуществующий объект
Если ты скомандуешь напечатать переменную, которую еще не объявил в коде выше, конвейер мгновенно остановится с критической ошибкой `NameError`.

### Защита Мастера
Никогда не вызывай `print()` вслепую для промежуточных вычислений внутри тяжелых циклов на миллионы строк — это парализует RAM и процессор. Вывод на экран — это ручной тормоз. Используй его точечно, только для фиксации ключевых вех скрипта.

## 3. Инструмент №2: Ячейка памяти — Переменная (`=`)

### Логика инструмента
Переменная — это именной ярлык, приклеенный к конкретному куску данных в оперативной памяти. Оператор `=` (присваивание) — это приказ: «Взять данные справа, положить их в RAM и наклеить на них ярлык с именем слева».

### Синтаксис
```python
object_name = "ЖК Центр"
fact_quantity = 150
```

### Физика работы
В отличие от жестких колонок в SQL, где тип данных прописан в схеме таблицы раз и навсегда, в Python действует **динамическая типизация**. Тип определяется автоматически по содержимому. Написал текст в кавычках — переменная стала текстовой. Написал целое число — числовой.

### КАРТА ВСЕЛЕННОЙ ТИПОВ ДАННЫХ (БАЗОВЫЕ ТИПЫ)
1. **`int` (Целые числа):** Числа без дробного хвоста (`1`, `25`, `-100`). Используются для штук, ID, счетчиков.
2. **`float` (Плавающие числа):** Числа с дробной частью (`15.5`, `0.003`, `-7.1`). В Python `float` — это аналог `REAL` / `FLOAT` в СУБД. Они имеют ту же уязвимость округления в двоичной системе.
3. **`str` (Строки / Текст):** Любые символы, зажатые в одинарные или двойные кавычки (`"Цемент"`, `'Марка М500'`).
4. **`bool` (Логический тип):** Имеет всего два состояния: `True` (Истина / `1`) или `False` (Ложь / `0`). Рождается в результате операций сравнения (сито фильтрации).

## 4. Инструмент №3: Базовая математика и работа со строками

### Логика инструмента
Математика в Python выполняется мгновенно над переменными в RAM. 

### Синтаксис
```python
# Числовая математика
total = fact + extra
delta = plan - fact
total_weight = volume * density
average_price = total_cost / total_orders

# Строковая математика (Сцепка / Конкатенация — аналог CONCAT)
full_name = material + " " + grade  # Результат: "Цемент М500"
```

### Физика работы
* Оператор `/` (деление) в Python 3 **всегда** возвращает тип `float`, даже если делишь `4 / 2` (результат будет `2.0`).
* Оператор `+` для строк работает как клей-момент (конкатенация), сшивая их в один монолитный текстовый объект.

### Капкан №1: Межклассовый конфликт (Попытка склеить текст и число)
Если ты попытаешься выполнить операцию `full_title = "Заказ №" + 105`, скрипт упадет со свистом с ошибкой `TypeError`. Python — язык с **жесткой типизацией**. Он не будет угадывать за тебя, что ты хочешь сделать: прибавить число к тексту или перевести текст в число.

### Защита Мастера
Перед склейкой всегда принудительно переплавляй числовые типы в текст (см. Инструмент №6 — `str()`).

### Капкан №2: Деление на ноль (`ZeroDivisionError`)
Если переменная в знаменателе примет значение `0` или `0.0`, выполнение всего кода будет заблокировано.

### Защита Мастера
Внедряй предохранители до момента деления с помощью логических условий (см. Главу 2) или специализированной обработки исключений.

## 5. Инструмент №4: Продвинутые операторы деления (`//` и `%`)

### Логика инструмента
1. `//` (Целочисленное деление) — рубит дробный хвост, оставляя только целую часть от деления (аналог `TRUNC` или переплавки в `INT`).
2. `%` (Остаток от деления) — вычисляет то, что осталось «за бортом» после деления нацело.

### Синтаксис
```python
hours = 25
days = hours // 24     # Результат: 1 (целые сутки)
rem_hours = hours % 24  # Результат: 1 (оставшийся час)
```

### Физика работы
Операции выполняются на уровне процессора. Остаток от деления `%` — это идеальный инструмент для проверки четности чисел (если `число % 2 == 0`, то оно четное) или для циклического распределения задач по пулу рабочих.

## 6. Инструмент №5: Переплавка типов — Инструменты `int()`, `float()`, `str()`

### Логика инструмента
Принудительное изменение типа данных переменной на ходу (полный аналог `CAST(col AS type)` или скоростного Postgres-маневра `col::type`).

### Синтаксис
```python
# Из текста в целое число
fact_count = int("150")

# Из целого числа в плавающее
weight_float = float(50)  # Результат: 50.0

# Из числа в текст для безопасной сцепки (CONCAT)
log_message = "Обработано строк: " + str(rows_processed)
```

### Физика работы
Python берет внутреннее значение объекта, проверяет его на совместимость с целевым форматом, создает в RAM новый объект требуемого типа и перенаправляет на него ярлык переменной.

### Капкан №1: Грязные данные при плавлении текста в число
Если в строке окажется хотя бы один нечисловой символ (например, `"150 шт"` или `"150 "` с невидимым пробелом на конце), инструмент `int()` выдаст критическую ошибку `ValueError`.

### Защита Мастера
Перед очисткой и плавлением текстовых выгрузок из ПТО всегда задействуй методы зачистки шума (см. Инструмент №7 — `.strip()`).

## 7. Инструмент №6: Зачистка текстового шума — Методы `.strip()`, `.lower()`, `.upper()`

### Логика инструмента
Полные аналоги «Команды ювелиров» из SQL (`TRIM`, `LOWER`, `UPPER`). Уничтожают паразитные пробелы по краям строк и приводят символы к единому регистру для точного сопоставления.

### Синтаксис
```python
raw_material = "  Арматура А500С  "

# Комплексная зачистка «в один проход» (Маневр Ювелира)
clean_material = raw_material.strip().lower()
# Результат: "арматура а500с"
```

### Физика работы
Метод `.strip()` сканирует текстовую строку с левого и правого конца, откусывая пробелы, знаки табуляции (`\t`) и символы переноса строки (`\n`). Метод `.lower()` переводит все буквы в нижний регистр. Оригинальная переменная `raw_material` не меняется — метод возвращает чистую виртуальную проекцию, которую мы фиксируем в новую переменную.

### Капкан №1: Попытка применить текстовый метод к числу
Если в переменной лежит тип `int` или `float`, вызов метода `.strip()` или `.lower()` приведет к падению скрипта с ошибкой `AttributeError`. У чисел нет таких методов.

### Защита Мастера
Если колонка грязная и тип данных не определен, сначала принудительно оберни объект в `str()`, а затем вызывай методы зачистки: `str(variable).strip().lower()`.

### Логика инструмента
`None` в Python — это абсолютное ничто, полная пустота (точный аналог `NULL` в SQL). Любая попытка выполнить математическую операцию или применить метод к `None` гарантированно убивает скрипт.

### Синтаксис

```python
fact_delivery = None
```

### Физика работы

### Защита Мастера
Перед проведением любых расчетов над импортированными данными всегда проверяй их на пустоту (инструменты проверки и аналог «Коалы» — `COALESCE` — будут детально развернуты в блоке управляющих конструкций и Pandas).

# Глава 2. ЛОГИЧЕСКОЕ СИТО И УПРАВЛЕНИЕ ПОТОКОМ: УСЛОВИЯ, СРАВНЕНИЯ И ПРЕДОХРАНИТЕЛИ ПУСТОТЫ

## 1. Введение в архитектуру: Что такое управляющие конструкции
В SQL мы управляем потоком данных глобально, отсекая строки в блоке `WHERE` или вешая ярлыки через `CASE WHEN`. В Python условия `if - elif - else` — это **стрелки на железнодорожных путях**. Скрипт движется сверху вниз, встречает развилку, проверяет логическое выражение на истинность (`True` / `False`) и направляет поток данных по нужной трубе вычислений.

---

## 2. Инструмент №1: Операторы сравнения (Рождение логического сита)

### Логика инструмента
Инструменты, которые берут два объекта, взвешивают их по заданному критерию и возвращают тип `bool` (`True` или `False`). 

### Карта операторов
* `==` — Проверка на точное равенство. (Внимание: один знак `=` — это приказ положить данные в память, два знака `==` — это вопрос «Они равны?»).
* `!=` — Проверка на неравенство (Истинно, если объекты НЕ совпадают).
* `>`, `<` — Больше / Меньше.
* `>=`, `<=` — Больше или равно / Меньше или равно.

### Синтаксис
```python
is_limit_exceeded = fact_volume > estimate_volume
```

## 3. Инструмент №2: Монолитное ветвление — `if - elif - else`

### Логика инструмента
Полный аналог конструкции `CASE WHEN ... THEN ... ELSE ... END` из SQL. Позволяет выполнять разные куски кода в зависимости от ситуации.

### Синтаксис
```python
if diameter == 89:
    category = "технологическая труба"
elif diameter == 108:
    category = "труба для теплосетей"
else:
    category = "труба_не_определена"
```

### Физика работы
* **Закон отступов (Жесткая дисциплина):** В Python нет оператора `END`, замыкающего блок, как в SQL. Границы блоков определяются **отступами (4 пробела или Tab)**. Всё, что написано с отступом вправо, выполняется только при истинности условия. Как только код возвращается на исходную вертикаль — блок закрыт.
* Система проверяет условия строго сверху вниз. Как только первое условие выдало `True`, Python заходит в этот цех, выполняет его код, а все остальные блоки `elif` и `else` внутри этой конструкции **игнорирует**, даже если они тоже истинны.

### Капкан №1: Потеря вертикали (Ошибка `IndentationError`)
Если ты нарушишь геометрию отступов (например, внутри одного `if` поставишь 4 пробела, а внутри `elif` — 3 пробела), компилятор откажется запускать скрипт.

### Защита Мастера
Используй современные редакторы (VS Code делает это на автопилоте). Всегда проверяй, чтобы блоки кода одного уровня вложенности стояли на одной строго выверенной вертикальной линии.

---

## 4. Инструмент №3: Логические связки — `and`, `or`, `not`

### Логика инструмента
Инструменты для объединения нескольких условий в один сложный фильтр (полные аналоги `AND`, `OR`, `NOT` в блоке `WHERE` или `WHEN`).

### Синтаксис
```python
# and — сужает сито (требует Истины от всех участников)
if status == "активен" and total_days > 30:
    print("Включить в табель")

# or — расширяет сито (достаточно Истины от одного участника)
if grade == "м500" or grade == "м400":
    print("Материал допущен")
```

### Физика работы
Python использует **ленивые вычисления (Short-circuit evaluation)** ради экономии процессора:
* В связке `and`, если первое условие оказалось `False`, система мгновенно бросает проверку и выдает `False`, даже не читая второе условие (зачем тратить ресурсы, если фильтр уже закрыт).
* В связке `or`, если первое условие оказалось `True`, система сразу заходит внутрь блока, игнорируя чтение последующих условий.

### Капкан №1: Приоритет вычислений без скобок
Оператор `and` имеет более высокий приоритет, чем `or` (как умножение важнее сложения в математике). Запрос `if cond_A or cond_B and cond_C` выполнится не так, как ты думаешь. Сначала свяжутся `B` и `C`, а потом к ним через «или» прицепится `A`.

### Защита Мастера
Изменяй порядок и защищай логику скобками, как в SQL: `if (cond_A or cond_B) and cond_C`. Скобки — это броня твоих условий.

## 5. Инструмент №4: Ловушка пустоты и аналог «Коалы» (`COALESCE`)

### Логика инструмента
В Python пустота `None` не умеет сравниваться с числами или текстом через стандартные операторы. Попытка написать `if None > 100` вызовет крах системы (`TypeError`). Прежде чем проводить расчеты над переменной, которая может прилететь пустой, её нужно проверить и подставить безопасную заплатку (0 или дефолтный текст).

### Синтаксис (Кустарный перехватчик пустот)
```python
# Настоящий аналог SELECT COALESCE(fact_delivery, 0)
if fact_delivery is None:
    safe_delivery = 0
else:
    safe_delivery = fact_delivery
```

### Физика работы
Для проверки на пустоту в Python используется специализированный оператор `is` и `is not`. Писать `if fact_delivery == None` — синтаксически неграмотно. Оператор `is` проверяет идентичность объектов на физическом уровне RAM. Поскольку `None` во всей системе существует в единственном экземпляре, мы спрашиваем: «Эта переменная указывает на ту самую системную пустоту?».

### Маневр Профи: Тернарный оператор (Коала в одну строку)
Чтобы не городить 4 строки кода ради одной заплатки, в Python используют компактную скоростную переплавку:
```python
safe_delivery = fact_delivery if fact_delivery is not None else 0
```
*Физика строки:* Запиши в `safe_delivery` значение `fact_delivery`, если оно НЕ является пустотой, иначе — запиши `0`.

```python
# =====================================================================

print("--- СТАРТ ЛОГИЧЕСКОГО СИТА ОПТИМИЗАТОРА ---")
```

# Глава 3. КОЛЛЕКЦИИ ДАННЫХ: ДИНАМИЧЕСКИЕ МАССИВЫ (LIST) И СВЯЗУЮЩИЕ СЛОВАРИ (DICT)

## 1. Введение в архитектуру: Зачем нужны коллекции?
До этого момента мы работали с одиночными переменными — это все равно что обрабатывать одну изолированную строку. Но на реальном производстве данные прилетают пакетами: списки сотрудников, массивы диаметров труб, справочники соответствия кодов ПТО. 
* **`List` (Список)** — это твоя вертикальная колонка в Excel или массив строк, где важен физический порядок.
* **`Dict` (Словарь)** — это инструмент создания неявных связей (маппинга), где по уникальному ключу (ID) система мгновенно вытягивает нужную характеристику.

---

## 2. Инструмент №1: Динамический конвейер — Список (`List`)

### Логика инструмента
Список — это упорядоченная последовательность элементов, зажатая в квадратные скобки `[]`. Он может хранить внутри себя объекты любых типов и динамически расширяться по ходу выполнения скрипта.

### Синтаксис
```python
diameters = [57, 89, 108, 219]
raw_data = ["Цемент", 50, 120.5, True]  # Разные типы в одном баке
```

### Физика работы: Индексация и Срезы (Аналог `LEFT`/`RIGHT`/`LIMIT`)
Каждый элемент в списке имеет свой порядковый номер — **индекс**. 
* **Внимание: В Python отсчет всегда начинается с `0`!** Первый элемент — `[0]`, второй — `[1]`.
* **Отрицательные индексы (Взгляд с тыла):** Индекс `[-1]` гарантированно вернет *последний* элемент в списке, `[-2]` — предпоследний. Это спасает, когда ты не знаешь точную длину массива.
* **Срезы (`Slices`):** Маневр `[start:stop]`. Извлечение куска массива. Правая граница `stop` **не включается** в результат (закон Python).

```python
pipes = ["Труба_57", "Труба_89", "Труба_108", "Труба_219"]
print(pipes[0])      # Результат: "Труба_57"
print(pipes[-1])     # Результат: "Труба_219" (последняя)
print(pipes[1:3])    # Результат: ["Труба_89", "Труба_108"] (индексы 1 и 2, 3-й не вошел)
```

### Модификация списка (Команды управления)
* `.append(element)` — Добавить элемент в самый конец списка (увеличить длину отчета вниз).
* `len(my_list)` — Посчитать общее количество живых строк в списке (полный аналог `COUNT(*)`).

### Капкан №1: Выход за границы массива (`IndexError`)
Если в твоем списке 4 элемента (индексы от 0 до 3), а ты скомандуешь выгрузить `pipes[4]`, конвейер моментально остановится из-за аварии `IndexError: list index out of range`.

### Защита Мастера
Перед точечным обращением по индексу всегда проверяй длину массива через `len()`, либо запрашивай безопасные срезы — срезы никогда не падают в ошибку, даже если ты укажешь границы с избытком, они просто вернут то, что выжило.

## 3. Инструмент №2: Мост неявных связей — Словарь (`Dict`)

### Логика инструмента
Словарь — это коллекция пар «ключ: значение», зажатая в фигурные скобки `{}`. Здесь нет физических индексов (0, 1, 2). Вместо этого ты сам назначаешь уникальный идентификатор (ключ), по которому система мгновенно, без перебора всего массива, находит значение. Это кустарная база данных в RAM.

### Синтаксис
```python
# Справочник лимитов (Маппинг "Материал -> Лимит")
limits_dict = {
    "цемент": 500,
    "арматура": 120,
    "песок": 2000
}
```

### Физика работы
Под капотом словаря лежит механизм **хэш-таблиц**. Доступ к значению по ключу происходит за фиксированное микро-время, независимо от того, лежит в словаре 3 строки или 3 000 000. 
* Ключ обязан быть уникальным и неизменяемым (обычно это текст `str` или число `int`).
* Значением может быть что угодно: число, текст, список или даже другой вложенный словарь.

```python
# Извлечение данных по ключу
current_limit = limits_dict["цемент"]  # Результат: 500

# Добавление новой пары или перезапись старой
limits_dict["щебень"] = 800   # Рождение новой строки
limits_dict["цемент"] = 600   # Перезапись существующего значения
```

### Капкан №1: Поиск несуществующего ключа (`KeyError`)
Если ты попытаешься вытащить лимит по материалу, которого нет в словаре (например, `limits_dict["кабель"]`), Python выбросит критическую ошибку `KeyError`. Весь расчет заморожен.

### Защита Мастера — Перехватчик пустот через метод `.get()`
Забудь про прямое обращение через квадратные скобки, если не уверен в чистоте справочника на 100%. Используй метод `.get(key, default)`. Это **абсолютный аналог «Коалы» (`COALESCE`) для словарей**. Если ключ найден — он вернет значение. Если не найден — он не упадет, а тихо вернет твою заплатку (например, 0).

```python
# Безопасный маневр: если кабеля нет в смете, лимит равен 0
safe_limit = limits_dict.get("кабель", 0)  # Результат: 0 (запрос спасен!)
```

# Глава 4. АВТОМАТИЗАЦИЯ ПОТОКА: ЦИКЛЫ FOR И КОМАНДЫ УПРАВЛЕНИЯ ПЕРЕБОРОМ

## 1. Введение в архитектуру: Переход на конвейерную сборку
До этого момента мы обрабатывали списки вручную по индексам — это все равно что ловить каждого рабочего на стройке лично ради одной подписи. Циклы в Python — это **автоматический промышленный конвейер**. Мы загружаем в начало трубы массив данных (список строк, табель, выгрузку из ПТО), задаем одно общее правило обработки, и система сама последовательно прогоняет через него каждый элемент сверху вниз до самой последней строчки.

* **В SQL** эту роль выполняет движок СУБД, обрабатывая миллионы строк невидимым внутренним циклом.
* **В Python** мы сами запускаем этот конвейер через синтаксический блок `for`.

---

## 2. Инструмент №1: Построчный переборщик — Цикл `for`

### Логика инструмента
Цикл `for` берет коллекцию данных (список, словарь или диапазон чисел), создает временную переменную-бегунка, берет первый элемент, выполняет над ним блок кода с отступом, возвращается наверх, берет второй элемент, и так до тех пор, пока элементы не закончатся.

### Синтаксис
```python
# Базовый перебор списка материалов
materials = ["Цемент", "Арматура", "Щебень"]

for material in materials:
    print("Проверка на складе: " + material)
```

### Физика работы
* `materials` — исходный массив строк (коллекция).
* `material` — имя временной переменной-бегунка. На каждом витке (итерации) цикла в нее автоматически залетает следующее значение из списка. Ты волен назвать ее как угодно (`item`, `row`, `x`), но имя должно отражать суть.
* **Жесткая дисциплина отступов:** Всё, что написано внутри цикла сдвинутым вправо (4 пробела или Tab), выполняется циклически. Как только код возвращается на исходную левую вертикаль — цикл завершен, конвейер остановился.

---

## 3. Инструмент №2: Генератор числовых диапазонов — `range()`

### Логика инструмента
Когда у тебя нет готового списка строк, но нужно выполнить действие ровно N раз (или сгенерировать сквозные индексы строк для табеля), используется встроенный счетчик `range()`.

### Синтаксис
```python
# Выполнить команду ровно 5 раз (счетчик выдаст числа от 0 до 4)
for i in range(5):
    print("Итерация конвейера №: " + str(i))
```

### Физика работы
Функция `range(start, stop, step)` генерирует последовательность целых чисел на лету, экономя оперативную память. Она не создает тяжелый список в RAM целиком.
* `range(5)` — выдаст `0, 1, 2, 3, 4`. Правая граница **никогда не включается** в результат (стандартный закон Python).
* `range(1, 6)` — выдаст числа от `1` до `5`.
* `range(0, 10, 2)` — выдаст четные числа с шагом 2: `0, 2, 4, 6, 8`.

---

## 4. Инструмент №3: Команды перенаправления потока — `continue` и `break`

### Логика инструмента
В процессе перебора миллионов строк скрипт может наткнуться на критическую аномалию. Нам нужны рычаги управления конвейером на ходу:
1. `continue` — Срочный сброс текущего витка. Скрипт мгновенно бросает выполнение оставшихся строк кода внизу цикла, возвращается наверх и берет *следующий* элемент из списка. (Аналог пропуска шума).
2. `break` — Аварийная остановка конвейера. Полный выход из цикла. Скрипт уничтожает цикл и идет дальше по коду ниже. (Аналог стоп-крана при критической ошибке).

### Синтаксис
```python
for num in range(1, 100):
    if num == 13:
        continue # Пропустить чертову дюжину, идти к числу 14
    if num == 50:
        break    # Дошли до половины, полностью заглушить конвейер
```

### Капкан №1: Бесконечные циклы и зависание RAM
В Python существует второй вид циклов — `while` (выполнять, пока условие истинно). Если в коде `while` допустить ошибку и не изменять условие внутри, скрипт уйдет в бесконечный цикл. Он намертво зажмет процессор и сожрет оперативную память, вызвав зависание верстака.

### Защита Мастера
В аналитике данных в 99% случаев используется строго цикл `for`. Он работает только в границах переданной ему коллекции (или диапазона `range`), поэтому физически не способен стать бесконечным. Забудь про `while` на этапе обработки витрин данных.

## 5. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (КОД ДЛЯ ТВОЕГО ВЕРСТАКА)

Скопируй этот код в свой файл `python_master.ipynb` и запусти ячейку. Мы собираем твой первый сквозной конвейер. Скрипт имитирует автоматический аудит суточного табеля рабочих СМР (Массив ООО «ФракДжет-Строй»). 

Конвейер выполняет комплексный маневр: построчно перебирает сотрудников, зачищает их грязные имена, перехватывает опасную пустоту `None`, считает общую сумму выплат, отсекает прогульщиков через `continue` и бьет по тормозам через `break`, если обнаруживает критическую ошибку в данных.

```python

# СЦЕНАРИЙ: Сквозной автоматический аудит грязного табеля ИТР и СМР
# =====================================================================

raw_timesheet = [
    [1, "  Иванов И.И. ", 8],

    [4, "КРИТ_ОШИБКА_БАЗЫ", -999], # Поломка логирования — стоп-кран!
    [5, "Николаев К.Д.", 8]       # Эта строка не должна обработаться из-за аварии выше

hourly_rate = 500  # Ставка: 500 рублей в час

valid_rows_count = 0 # Счетчик успешно обработанных сотрудников (Аналог COUNT)

for row in raw_timesheet:
    row_id = row[0]
    raw_name = row[1]
    raw_hours = row[2]
    
    # ШАГ 1: Ювелирная зачистка текстового шума
    clean_name = raw_name.strip().upper()
    
    # ШАГ 2: Защита Мастера — Проверка на критическую аварию логов
    if "ОШИБКА" in clean_name or (raw_hours is not None and raw_hours < 0):
        print(f"АВАРИЙНЫЙ СТОП-КРАН: Строка №{row_id} повреждена ({clean_name}). Конвейер заблокирован!")
        break # Полная остановка системы, строки ниже не обрабатываются
        
    # ШАГ 3: Перехватчик пустоты (Аналог "Коалы" / COALESCE)
    # Если прилетел None — временно считаем, что человек отработал 0 часов
    hours = raw_hours if raw_hours is not None else 0
    
    # ШАГ 4: Отсечение нецелевых строк (Пропускаем тех, кто не работал)
    if hours == 0:
        print(f"Строка №{row_id}: Рабочий {clean_name} отработал 0 часов. Пропуск строки.")
        continue # Скрипт бросает этот виток и летит наверх к следующему рабочему
        
    # ШАГ 5: Бизнес-логика (Расчет денег и накопление итогов)
    salary = hours * hourly_rate
    total_payout = total_payout + salary # Накапливаем сумму (SUM)
    valid_rows_count = valid_rows_count + 1 # Наращиваем счетчик (COUNT)
    
    print(f"Строка №{row_id}: Рабочий {clean_name} проверен. Выплата: {salary} руб.")

print("-" * 50)
print(f"ИТОГ КОНВЕЙЕРА:")
print(f"Успешно обработано сотрудников (COUNT): {valid_rows_count}")
print(f"Общая сумма к выплате на участок (SUM): {total_payout} руб.")
print("--- СИНХРОНИЗАЦИЯ ПОТОКА ЗАВЕРШЕНА ---")
```

## 5. Инструмент №3: Приборы разведки и аудита качества — `.info()`, `.describe()`, `.head()`

* `df.describe()` — Быстрая математическая статистика по всем числовым колонкам. Выдает среднее значение (аналог `AVG`), минимум (`MIN`), максимум (`MAX`) и распределение данных.


Если в числовой колонке (например, объемы поставок) в исходном Excel-файле затесалась хотя бы одна текстовая ячейка (например, слово `"нет"` или `"ошибка"`), Pandas принудительно присвоит ВСЕЙ этой колонке тип данных `object` (текстовый). # Глава 6. ХИРУРГИЧЕСКИЙ ОТСЕВ И ОЧИСТКА: ФИЛЬТРАЦИЯ СТРОК (WHERE) И СЕЙВ ОТ ПУСТОТ (COALESCE) В PANDAS

## 1. Введение в архитектуру: Операции над осями в RAM
В СУБД мы отсекаем строки на дальних подступах к оперативной памяти через блок `WHERE`, чтобы процессор работал только со «скульптурой». В Pandas данные уже лежат в RAM. Наша задача — хирургически срезать лишний «камень», применяя быстрые векторные маски, зачистить скрытые пустоты и принудительно переплавить сломанные типы колонок в чистые числа.

## 2. Инструмент №1: Фильтрация строк через Логические маски — `df[маска]`

### Логика инструмента
Абсолютный аналог блока `WHERE` в SQL. Мы задаем условие к конкретной колонке. Pandas сканирует её, проверяет каждую ячейку и формирует невидимый логический щит (маску из `True` и `False`). Затем этот щит прижимается к датафрейму, и через него проскакивают только те строки, где условие выдало `True`.

### Синтаксис
```python
# Настоящий аналог: WHERE object_name == 'ЖК Центр'
mask = df["object_name"] == "ЖК Центр"
df_filtered = df[mask]

# Скоростной маневр в одну строку на автопилоте рук:
df_filtered = df[df["object_name"] == "ЖК Центр"]
```

### Физика работы
Операция выполняется векторизованно на уровне языка C, без медленных циклов Python. Исходный датафрейм `df` не разрушается — система создает в RAM его очищенную виртуальную проекцию `df_filtered`.

### Сочетание условий (Связки AND / OR)
В Pandas вместо слов `and` и `or` внутри фильтров используются жесткие побитовые операторы. **Каждое отдельное условие обязательно зажимается в круглые скобки!**
* `&` — Побитовое И (аналог `AND`). Сужает сито вычислений.
* `|` — Побитовое ИЛИ (аналог `OR`). Расширяет сито вычислений.

```python
# Аналог: WHERE (diameter == 89) AND (status == 'Принято')
df_clean = df[(df["diameter"] == 89) & (df["status"] == "Принято")]
```

## 3. Инструмент №2: Охотник за пустотой и Заплатка «Коалы» — `.isna()` и `.fillna()`

### Логика инструмента
В Pandas системная пустота (`NULL`) обозначается как `NaN` (Not a Number) для чисел или `None` для текста. 
1. Метод `.isna()` — детектор дыр (полный аналог `IS NULL` в SQL).
2. Метод `.fillna(value)` — перехватчик пустот и установка заплатки (абсолютный аналог оператора `COALESCE`).

### Синтаксис
```python
# 1. Найти все строки с дырами (Аналог: WHERE volume IS NULL)
df_holes = df[df["volume"].isna()]

# 2. Маневр "Коалы": Заплатка на дыру (Аналог: COALESCE(volume, 0.0))
df["volume"] = df["volume"].fillna(0.0)
```

### Физика работы
Метод `.fillna(0.0)` сканирует указанную колонку. Как только система натыкается на метку пустоты `NaN`, она выжигает её и вставляет на это место твое безопасное значение `0.0`. Это спасает будущую математику от каскадного обнуления.

## 4. Инструмент №3: Принудительная переплавка типов — `.astype()` и `pd.to_numeric()`

### Логика инструмента
Исправление сломанных типов колонок (полный аналог `CAST(col AS type)` или скоростного маневра `col::type`).

### Синтаксис
```python
# Классическая переплавка, если в колонке чистый текст-число
df["line_id"] = df["line_id"].astype("int64")

# Тяжелая артиллерия для грязных колонок с мусором:
df["fact_tons"] = pd.to_numeric(df["fact_tons"], errors="coerce")
```

### Физика работы и Капкан №1: Падение при плавке грязного текста
Если ты применишь метод `.astype("float64")` к колонке, где среди чисел сидит текстовый мусор (слово `"ошибка"` или `"нет данных"`), скрипт упадет. Конвейер заблокирован.

### Защита Мастера — Маневр с параметром `errors='coerce'`
Используй специализированный инструмент `pd.to_numeric(column, errors='coerce')`. 
* Физика маневра: Pandas начинает плавить элементы колонки в числа. Если он встречает чистую строку-число `"50.5"`, он превращает её во `50.5`. Если он натыкается на текстовый мусор `"грязный_шум"`, параметр `errors='coerce'` **принудительно превращает этот мусор в безопасную пустоту `NaN`**. Текст уничтожен. После этого мы мгновенно закрываем зазор методом `.fillna(0.0)`. Колонка спасена и переведена в числовой формат!

## 5. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (КОД ДЛЯ ТВОЕГО ВЕРСТАКА)

Этот код решает критические баги, которые мы обнаружили при разведке в Главе 5. Скопируй его в свой `python_master.ipynb` и запусти ячейку. 

Скрипт проводит хирургический ремонт таблицы ПТО: полностью ликвидирует текстовый мусор в тоннаже, преобразует типы данных, перехватывает пустые даты через маневр «Коалы» и отсекает лишний «камень» фильтрацией по условиям.

```python
# =====================================================================
# СЦЕНАРИЙ: Хирургическое лечение, переплавка типов и фильтрация датафрейма ПТО
# =====================================================================

import pandas as pd

# Загружаем ту самую грязную структуру из Главе 5
dirty_pto_data = {
    "line_id":,
    "object_name": ["ЖК Центр", "  ЖК Центр  ", "Объект Восточный", "ЖК Центр", "Объект Восточный"],
    "material_type": ["ЦЕМЕНТ М500", "АРМАТУРА А500С", "БЕТОН В25", "ЦЕМЕНТ М500", "АРМАТУРА А500С"],
    "fact_tons": [50.5, "120.0", 300.0, "грязный_шум", 85.2], 
    "delivery_date": ["2026-05-10", "2026-05-11", None, "2026-05-13", "2026-05-14"] 
}
df_pto = pd.DataFrame(dirty_pto_data)

print("--- СТАРТ РЕФАКТОРИНГА И ОЧИСТКИ ТАБЛИЦЫ ---")

# ШАГ 1: Лечение сломанной колонки 'fact_tons' (Маневр Мастера с errors='coerce')
# Мусор "грязный_шум" принудительно плавится в NaN
df_pto["fact_tons"] = pd.to_numeric(df_pto["fact_tons"], errors="coerce")

# Мгновенно закрываем образовавшуюся дыру заплаткой 0.0 (Аналог COALESCE)
df_pto["fact_tons"] = df_pto["fact_tons"].fillna(0.0)


# ШАГ 2: Заплатка на пустую дату доставки (Маневр "Коалы")
# Если дата утеряна — пишем системный маркер 'НЕТ_ДАТЫ'
df_pto["delivery_date"] = df_pto["delivery_date"].fillna("НЕТ_ДАТЫ")


# ШАГ 3: Ювелирная зачистка текстового шума в названиях объектов
# Чтобы строки склеивались идеально, убираем пробелы по краям
df_pto["object_name"] = df_pto["object_name"].str.strip()


# КОНТРОЛЬНЫЙ АУДИТ ПОСЛЕ ЛЕЧЕНИЯ
print("\n[АУДИТ] Структура таблицы после рефакторинга:")
df_pto.info()
print("-" * 60)


# ШАГ 4: Хирургический отсев строк (Аналог блока WHERE с двумя условиями)
# Задача: Оставить в отчете только поставки на 'ЖК Центр', где объем строго больше 0
mask_where = (df_pto["object_name"] == "ЖК Центр") & (df_pto["fact_tons"] > 0)
df_final_report = df_pto[mask_where]


print("\n[ФИНАЛЬНЫЙ СЛУЖЕБНЫЙ ОТЧЕТ] Скульптура данных готова к отгрузке:")
print(df_final_report)
print("--- СИНХРОНИЗАЦИЯ МОДУЛЯ ОЧИСТКИ ЗАВЕРШЕНА ---")
```

# Глава 7. АГРЕГАЦИЯ И СТЫКОВКА МИРОВ: МЕТОД .groupby() И СШИВКА ТАБЛИЦ (JOIN) В PANDAS

## 1. Введение в архитектуру: Схлопывание групп и наведение мостов в RAM
До этого момента мы фильтровали и чистили плоские массивы данных. Теперь мы переходим к аналитическим операциям высшего порядка. 
* **Группировка (`.groupby()`)** — это точный аналог оператора `GROUP BY` в SQL. Мы схлопываем тысячи одноименных строк в компактные аналитические узлы, чтобы посчитать внутри них финансовые и производственные итоги (SUM, COUNT, AVG).
* **Сшивка таблиц (`pd.merge()`)** — это наведение виртуальных мостов между изолированными датафреймами. Полный аналог операторов `JOIN ON` в СУБД.

---

## 2. Инструмент №1: Схлопывание строк в группы — `.groupby()`

### Логика инструмента
Метод объединяет строки с одинаковыми значениями в указанном столбце в единые группы и применяет к числовым полям функции агрегации (сумма, среднее, количество).

### Синтаксис
```python
# Настоящий аналог: SELECT object_name, SUM(fact_tons) FROM table GROUP BY object_name
df_grouped = df.groupby("object_name")["fact_tons"].sum().reset_index()
```

### Физика работы
1. `df.groupby("object_name")` — приказ системе изолировать строки по уникальным именам объектов.
2. `["fact_tons"]` — указание целевой колонки, над которой будет производиться математика.
3. `.sum()` — функция агрегации (также доступны `.count()`, `.mean()`, `.max()`, `.min()`).
4. **`.reset_index()` (Закон Мастера):** По умолчанию Pandas превращает колонку группировки в жирный индекс строки. Метод `.reset_index()` принудительно возвращает таблицу в стандартный плоский вид, делая `object_name` обычной колонкой. Забудь про этот метод — и ты не сможешь отфильтровать или сшить полученный результат дальше по конвейеру.

### Комбинированные группы (Группы внутри групп)
Если нужно сгруппировать по нескольким осям сразу (аналог `GROUP BY col1, col2`), имена колонок передаются списком в квадратных скобках `[]`:
```python
df_double_group = df.groupby(["object_name", "material_type"])["fact_tons"].sum().reset_index()
```

---

## 3. Инструмент №2: Сшивка миров таблиц — `pd.merge()`

### Логика инструмента
Инструмент для горизонтального соединения двух независимых датафреймов по общим ключевым столбцам (ID, шифры объектов, уникальные имена).

### Синтаксис
```python
# Настоящий аналог: FROM df_left LEFT JOIN df_right ON df_left.id = df_right.id
df_joined = pd.merge(df_left, df_right, on="material_id", how="left")
```

### Физика работы
* `df_left` и `df_right` — левая (таблица-лидер) и правая таблицы.
* `on="material_id"` — общая точка опоры (ключ сцепки). Если имена ключей в таблицах разные, используют параметры `left_on="id_левой"` и `right_on="id_правой"`.
* `how="left"` — режим стыковки (мост). Доступны 4 классических режима SQL:
  * `'left'` — удерживает контроль: все строки левой таблицы выживут, а не нашедшие пару правые ячейки заполнятся пустотой `NaN`.
  * `'inner'` — мост пар: оставит только стопроцентную взаимность (строки, которые нашли пару в обоих мирах).
  * `'right'` — удерживает правую таблицу.
  * `'outer'` — объединяет всё, заполняя пустоты с обеих сторон.

### Капкан №1: Раздувание RAM при дубликатах в ключах (`Many-to-Many`)
Если в обеих таблицах по точке опоры `on` ключи повторяются многократно, `pd.merge()` начнет перемножать строки, генерируя гигантское количество дубликатов. Это мгновенно раздует отчет и может полностью сожрать оперативную память (RAM), обрушив скрипт.

### Защита Мастера
Перед сшивкой всегда проверяй качество ключей в правой таблице, либо предварительно «суши» и спрессовывай её через быстрый `.groupby()`, превращая ключи в уникальные строки до момента захода в мост `pd.merge()`.

## 4. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (КОД ДЛЯ ТВОЕГО ВЕРСТАКА)

Этот код моделирует реальный бизнес-процесс «Выполнения». Мы берем таблицу фактических поставок с линии СМР, спрессовываем её через группировку по объектам и материалам, а затем сшиваем по неявной связи со справочником сметных лимитов, вычисляя остатки лимитов и перерасходы.

```python
# =====================================================================
# СЦЕНАРИЙ: Агрегация поставок СМР и сшивка со сметой для выявления перерасходов
# =====================================================================

import pandas as pd

print("--- СИНХРОНИЗАЦИЯ МОДУЛЯ АГРЕГАЦИИ И СШИВКИ ---")

# Таблица №1: Лог фактических поставок на участки (Грязный оффлайн-массив)
fact_delivery_data = {
    "object": ["ЖК Центр", "ЖК Центр", "Объект Восточный", "ЖК Центр", "Объект Восточный"],
    "material": ["цемент м500", "арматура а500с", "бетон в25", "цемент м500", "арматура а500с"],
    "tons": [50.0, 120.0, 300.0, 45.5, 15.0] # Несколько строк по одному материалу на один объект
}
df_fact = pd.DataFrame(fact_delivery_data)

# Таблица №2: Справочник жестких сметных лимитов (Уникальные плановые показатели)
estimate_data = {
    "object": ["ЖК Центр", "ЖК Центр", "Объект Восточный"],
    "material": ["цемент м500", "арматура а500с", "бетон в25"],
    "limit_tons": [100.0, 100.0, 250.0] # Лимиты на объекты
}
df_estimate = pd.DataFrame(estimate_data)

# ---------------------------------------------------------------------
# ШАГ 1: Агрегация факта (Схлопываем строки до уникальных пар Объект-Материал)
# ---------------------------------------------------------------------
# Настоящий аналог GROUP BY object, material -> SUM(tons)
df_fact_aggr = df_fact.groupby(["object", "material"])["tons"].sum().reset_index()

print("[ШАГ 1] Фактические поставки после схлопывания групп:")
print(df_fact_aggr)
print("-" * 60)

# ---------------------------------------------------------------------
# ШАГ 2: Сшивка миров (Строим Левый мост по составному ключу — по двум опорам)
# ---------------------------------------------------------------------
# Связываем факт со сметой одновременно по Объекту И Материалу (Composite JOIN)
df_report = pd.merge(df_fact_aggr, df_estimate, on=["object", "material"], how="left")

print("[ШАГ 2] Результат сшивки Факта и Сметных лимитов (Левый мост):")
print(df_report)
print("-" * 60)

# ---------------------------------------------------------------------
# ШАГ 3: Бизнес-логика (Прямая векторная математика по колонкам в RAM)
# ---------------------------------------------------------------------
# Защищаем сметный лимит от пустот: если материала не было в смете — ставим 0.0
df_report["limit_tons"] = df_report["limit_tons"].fillna(0.0)

# Вычисляем остаток лимита (Векторная операция: столбец минус столбец)
df_report["delta_tons"] = df_report["limit_tons"] - df_report["tons"]

print("[ФИНАЛЬНЫЙ АНАЛИТИЧЕСКИЙ ОТЧЕТ ПТО] Витрина контроля лимитов:")
print(df_report)
print("\n--- МАНЕВР ВЫПОЛНЕН. ДАННЫЕ СТАБИЛИЗИРОВАНЫ ---")
```
# Глава 8. СИНХРОНИЗАЦИЯ С ХРАНИЛИЩЕМ: БИБЛИОТЕКА SQLALCHEMY И АВТОМАТИЧЕСКАЯ ОТГРУЗКА В СУБД

## 1. Введение в архитектуру: Закрытие конвейера обработки данных
До этого момента мы импортировали, чистили, фильтровали и агрегировали данные внутри изолированного бункера оперативной памяти (RAM) с помощью Pandas. Но RAM энергозависима: скрипт завершил работу — данные испарились. Финальная стадия любого промышленного конвейера аналитика — **перманентная консервация результатов**. 

Вместо того чтобы вручную выгружать таблицы в Excel-файлы и захламлять жесткий диск, мы наводим прямой скоростной мост между Python и промышленной базой данных СУБД (PostgreSQL / MySQL) с помощью библиотеки **SQLAlchemy**.

* **В ручном режиме:** Аналитик сохраняет CSV, открывает клиент СУБД (DBeaver/pgAdmin), создает таблицу руками и импортирует файл.
* **В автоматическом режиме:** Скрипт Python одной командой создаёт таблицу нужной структуры в базе данных и заливает туда миллионы строк за секунды.

---

## 2. Инструмент №1: Движок подключения — `create_engine()`

### Логика инструмента
Инструмент библиотеки SQLAlchemy, который создает постоянную точку связи («туннель») между скриптом Python и сервером базы данных. Через этот туннель транслируются SQL-запросы и передаются пакеты строк.

### Синтаксис
```python
from sqlalchemy import create_engine

# Сборка строки подключения (URL-адрес базы данных)
# Формат: СУБД://Логин:Пароль@Сервер:Порт/Имя_Базы
db_url = "postgresql://pi_operator:Secr3tPass@192.168.1.50:5432/production_db"

# Инициализация движка-туннеля
engine = create_engine(db_url)
```

### Физика работы
При вызове `create_engine()` Python не подключается к базе мгновенно. Он лишь формирует чертеж подключения в RAM. Физический контакт и аутентификация логина/пароля произойдут автоматически, когда мы попытаемся отправить первую порцию данных.

---

## 3. Инструмент №2: Автоматический десант в СУБД — метод `.to_sql()`

### Логика инструмента
Встроенный в Pandas метод, который берет готовый `DataFrame` из оперативной памяти ПК, автоматически сканирует его структуру (названия колонок и типы данных), создает идентичную физическую таблицу на сервере СУБД и осуществляет пакетную заливку строк.

### Синтаксис
```python
# Настоящий приказ: "Сформировать таблицу в СУБД и залить туда данные из RAM"
df_report.to_sql(
    name="vitrina_payout_smr",  # Физическое имя новой таблицы в базе данных
    con=engine,                  # Наш созданный движок-туннель подключения
    if_exists="replace",        # Стратегия поведения, если таблица уже есть
    index=False,                # Отрубить технический индекс Pandas (Закон!)
    chunksize=10000             # Размер пакета для скоростной загрузки
)
```

### Физика работы: Стратегии параметра `if_exists`
Это рубильник, определяющий поведение системы при столкновении с существующей таблицей в базе:
* `if_exists='fail'` — По умолчанию. Если таблица с таким именем уже создана в СУБД, скрипт мгновенно прекратит работу с ошибкой `ValueError`, защищая данные от перезаписи.
* `if_exists='replace'` — Жесткий приказ «Снести и пересоздать». Система выполняет в базе команду `DROP TABLE`, уничтожая старый архив, создает чистую структуру заново и заливает свежие данные.
* `if_exists='append'` — Режим дозаписи (хронология). Система оставляет структуру старой таблицы нетронутой и просто приклеивает новые строки вниз (аналог вертикального моста `UNION ALL` или команды `INSERT INTO`).

### Капкан №1: Попытка залить DataFrame с грязными именами колонок
Если в твоем `DataFrame` названия колонок написаны на русском языке, содержат пробелы или спецсимволы (например, `"Объем поставки (тн)"`), Pandas честно попытается создать таблицу с такими именами в СУБД. Но если для PostgreSQL русские буквы в кавычках допустимы, то при попытке сделать к этой таблице SELECT в СУБД, ты столкнешься с постоянной необходимостью зажимать имена в двойные кавычки. Скрипты автоматизации начнут ломаться.

### Защита Мастера
Перед вызовом метода `.to_sql()` всегда принудительно переводи все заголовки колонок датафрейма в нижний регистр, заменяй пробелы на нижнее подчеркивание `_` и избавляйся от кириллицы. Колонки должны выглядеть монолитно: `volume_tons`, `line_id`, `delivery_status`.

### Капкан №2: Зависание сервера при построчной заливке больших массивов
Если в твоем датафрейме 1 000 000 строк, и ты вызовешь метод `.to_sql()` без дополнительных настроек, Pandas начнет отправлять на сервер базы данных миллион отдельных команд `INSERT`. Сетевой трафик захлебнется, процессор СУБД уйдет в 100% нагрузку, и заливка растянется на часы.

### Защита Мастера — Пакетная нарезка через `chunksize`
Всегда явно прописывай параметр `chunksize=10000` внутри метода `.to_sql()`. Это приказ системе: «Нарезать массив в RAM на пакеты по 10 000 строк и отправлять их в базу единым залпом». Скорость десантирования данных вырастает в сотни раз.

## 4. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (КОД ДЛЯ ТВОЕГО ВЕРСТАКА)

Этот код полностью автономен и готов к тестам без развертывания тяжелого внешнего сервера PostgreSQL. Мы используем встроенный в Python легковесный движок **SQLite**, который создает полноценную базу данных прямо в файле на твоем жестком диске. 

Скрипт имитирует завершение рабочего дня аналитика: берет очищенную в Pandas витрину данных табеля ПТО и автоматически выгружает её на вечное хранение в локальную СУБД, формируя готовую физическую таблицу.

```python
# =====================================================================
# СЦЕНАРИЙ: Автоматическая консервация аналитической витрины ПТО в СУБД
# =====================================================================

import pandas as pd
from sqlalchemy import create_engine

print("--- ИНИЦИАЛИЗАЦИЯ ТРАНСПОРТНОГО КОНВЕЙЕРА В СУБД ---")

# 1. Готовый очищенный DataFrame в RAM, прошедший все стадии рефакторинга
clean_vitrina_data = {
    "worker_id":,
    "worker_name": ["ИВАНОВ И.И.", "СИДОРОВ А.П.", "НИКОЛАЕВ К.Д.", "ПЕТРОВ В.С."],
    "hours_fact":,
    "payout_rub": [4000.0, 3500.0, 4000.0, 2000.0],
    "sync_status": ["PROCESSED", "PROCESSED", "PROCESSED", "PROCESSED"]
}
df_to_save = pd.DataFrame(clean_vitrina_data)

# 2. Создаем движок-подключение к локальной базе данных SQLite
# Файл 'pto_archive.db' автоматически родится в папке со скриптом
engine = create_engine("sqlite:///pto_archive.db")

print("[СИНХРОНИЗАЦИЯ] Наведение моста с базой pto_archive.db выполнено успешно.")

# 3. Десантируем данные из оперативной памяти в физическую таблицу СУБД
# Применяем стратегию 'replace': если таблица существовала — сносим старую и пишем свежую
df_to_save.to_sql(
    name="vitrina_daily_payout", 
    con=engine, 
    if_exists="replace", 
    index=False, 
    chunksize=5000
)

print("[УСПЕХ] Данные успешно законсервированы в таблицу 'vitrina_daily_payout'.")
print("-" * 60)

# 4. КОНТРОЛЬНАЯ ПРОВЕРКА: Проверяем, физически ли легли данные в базу?
# Отправляем прямой SQL-запрос в созданную базу данных через инструмент Pandas pd.read_sql
print("[АУДИТ ХРАНИЛИЩА] Делаем контрольный SELECT из СУБД для проверки:")

df_check_from_db = pd.read_sql("SELECT * FROM vitrina_daily_payout WHERE hours_fact > 5", con=engine)

print(df_check_from_db)
print("\n--- ТРАНСПОРТНЫЙ КОНВЕЙЕР УСПЕШНО ЗАКРЫТ. СИСТЕМА СТАБИЛЬНА ---")
```

---
